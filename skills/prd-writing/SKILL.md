# PRD（製品要件定義書）作成

ユーザーの要件をヒアリングし、開発チームが実装可能な形式のPRDを作成する。

## Purpose

製品要件を明確化し、ステークホルダー間の認識を統一。開発チームが正しく実装できるドキュメントを作成する。

## Key Concepts

### PRDの構成要素

1. **概要**: 製品・機能の目的
2. **背景**: なぜ必要か、解決する課題、**現状の問題と理想の状態**
3. **ユーザーストーリー**: 誰が・何を・なぜ
4. **機能要件**: 具体的な機能
5. **非機能要件**: パフォーマンス、セキュリティ等
6. **受入基準**: 完了の定義
7. **設計判断と代替案**: **なぜこのアプローチにしたか、検討したが採用しなかった案**
8. **スコープ外**: 今回対象外の内容・**なぜスコープ外としたか**
9. **制約条件**: 技術的・ビジネス的・組織的制約
10. **マイルストーン**: タイムライン
11. **用語集**: ドメイン用語・略語・固有名詞の定義

### 10年後耐久性の原則（CRITICAL）

**「10年後に読んでも、なぜこの機能が必要で、なぜこの仕様にしたかが分かる」PRDを書く。**

- ビジネス上の背景と課題を明確に（「売上が〇〇だから」ではなく「〇〇という課題があるため」）
- 技術選定・設計判断の理由を残す
- 却下した代替案も記録（将来同じ検討をせずに済む）
- ドメイン用語を定義する（業界特有の言葉を知らない人でも読めるように）

### ユーザーストーリー形式

```
[ペルソナ]として、
[機能・行動]したい、
なぜなら[理由・価値]だからだ。
```

### 受入基準（Gherkin形式）

```
Given [前提条件]
When [アクション]
Then [期待結果]
```

## Application

### Step 1: ヒアリング

以下を確認:
1. 何を作りたいか？（機能の概要）
2. 誰のための機能か？（ターゲットユーザー）
3. なぜ必要か？（背景・課題）
4. 成功の定義は？（KPI・目標）
5. 制約はあるか？（期限、技術、予算）

### Step 2: ユーザーストーリー作成

ヒアリング結果からユーザーストーリーを作成:
- 主要なユースケースを洗い出し
- 優先度を設定（Must/Should/Could/Won't）

### Step 3: PRDドラフト作成

```markdown
# [機能名] PRD

> 作成日: YYYY-MM-DD ／ 作成者: [名前（役割）]
> YYYY年MM月時点の要件。ビジネス環境の変化に応じて更新が必要。

## 概要
[1-2文で機能を説明。この機能が何をするものか、前提知識ゼロの人に伝わるように]

## 背景・目的
### 現状の課題
[どのような問題が起きているか。具体的な数値・事例があれば記載]

### なぜ今このタイミングか
[なぜ今この機能が必要なのか。きっかけ・ビジネス上の理由]

### 期待される効果
[機能によって得られる価値。定量的に示せるものは数値で]

## ターゲットユーザー
[ペルソナの説明。どんな人が、どんな場面で、どんな困りごとを持っているか]

## ユーザーストーリー

### US-1: [ストーリー名]
**ユーザー**: [ペルソナ]
**ストーリー**: [ペルソナ]として、[機能]したい、なぜなら[理由]だからだ。

**受入基準**:
- Given [前提条件]
  When [アクション]
  Then [期待結果]

**優先度**: Must/Should/Could

### US-2: [ストーリー名]
...

## 機能要件

### 必須要件（Must）
- [ ] 要件1
- [ ] 要件2

### 推奨要件（Should）
- [ ] 要件1

### 希望要件（Could）
- [ ] 要件1

## 非機能要件

### パフォーマンス
- [要件]

### セキュリティ
- [要件]

### ユーザビリティ
- [要件]

## 設計判断と代替案
| 判断内容 | 採用した案 | 理由 | 検討したが不採用とした案 | 不採用の理由 |
|---------|-----------|------|---------------------|------------|
| [判断1] | [採用案] | [なぜ] | [代替案] | [なぜ不採用] |

## 制約条件
- **技術的制約**: [既存システムとの互換性、使用可能な技術等]
- **ビジネス的制約**: [予算、期限、人員等]
- **組織的制約**: [承認プロセス、他チームとの依存関係等]

## スコープ外
| 対象外の内容 | なぜスコープ外としたか |
|------------|---------------------|
| [内容1] | [理由] |
| [内容2] | [理由] |

## マイルストーン
| マイルストーン | 予定日 | 内容 |
|--------------|-------|------|
| 設計完了 | YYYY-MM-DD | |
| 開発完了 | YYYY-MM-DD | |
| リリース | YYYY-MM-DD | |

## 用語集（該当がある場合）
| 用語 | 説明 |
|------|------|
| [ドメイン用語・略語] | [定義と文脈] |

## 関連ドキュメント
- [デザインドキュメント]
- [技術仕様書]

## 更新履歴
| 日付 | 更新者 | 内容 |
|------|-------|------|
| YYYY-MM-DD | [名前] | 初版作成 |
```

### Step 4: レビュー

- 要件の抜け漏れがないか
- 受入基準が明確か
- スコープが適切か
- ステークホルダーに確認

### Step 5: 知見保存

PRD作成のプロセスで得た知見をmemories/に保存。

## Examples

### 例: ユーザー登録機能のPRD

```markdown
# ユーザー登録機能 PRD

## 概要
新規ユーザーがメールアドレスとパスワードでアカウントを作成できる機能。

## 背景・目的
### 解決する課題
現状、サービス利用にはアカウントが必要だが、登録フローがない。

### 期待される効果
- 新規ユーザー獲得
- ユーザーデータの蓄積

## ユーザーストーリー

### US-1: メール登録
**ユーザー**: 新規訪問者
**ストーリー**: 新規訪問者として、メールアドレスで登録したい、なぜならすぐにサービスを使いたいからだ。

**受入基準**:
- Given ユーザーが登録ページにいる
  When メールアドレスとパスワードを入力して送信
  Then アカウントが作成され、確認メールが送信される

**優先度**: Must
```

## Common Pitfalls

| 落とし穴 | 結果 | 対策 |
|---------|------|------|
| 受入基準が曖昧 | 「完了」の認識がズレる | Gherkin形式で具体的に |
| スコープが広すぎ | 期限に間に合わない | スコープ外を明記（理由付き） |
| ユーザー視点の欠如 | 使いにくい機能になる | ユーザーストーリーから始める |
| 「なぜ」の欠如 | 後から仕様の意図が分からない | 設計判断と代替案セクションで記録 |
| 制約条件の省略 | 実装時に想定外の壁にぶつかる | 技術・ビジネス・組織の制約を明記 |
| ドメイン用語の未定義 | 開発者が仕様を誤解する | 用語集セクションで定義 |
