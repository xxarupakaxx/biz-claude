# 日報テンプレート

## 出力フォーマット

```markdown
# 日報 YYYY-MM-DD（曜日）

## 今日やったこと

### [タスク名]
- 実施内容の概要
- 主な作業内容

### [タスク名2]
- ...

## 成果・完了したもの

- [x] 完了したタスクや成果物（ドキュメント、レポート等）
- [x] ...

## 課題・問題点

- 遭遇した問題や未解決の課題
- ブロッカーとなっている事項
- （なければ「特になし」）

## 明日のTODO

- [ ] 未完了タスク
- [ ] 新たに発生したタスク
- [ ] 後回しにした項目（99_history.mdから抽出）

## ふりかえり（KPT）

### Keep（良かったこと・続けたいこと）
- 効率的だったプロセスや良い判断

### Problem（問題・改善が必要なこと）
- 時間がかかった作業、非効率だった点

### Try（次に試すこと）
- 改善案、新しいアプローチ
```

## 抽出ルール

### 05_log.md からの抽出（必須）

- `**ユーザー指示:**` ブロック → タスクの目的を把握
- `**レスポンス:**` ブロック → 実施内容を抽出
- Phase情報があれば進行状況を判断

### 20_survey.md からの抽出

- 調査結果の要約 → 「やったこと」セクションに反映
- 発見事項 → KPTのKeepに反映可能

### 30_plan.md からの抽出

- 計画の概要 → タスクの全体像を把握
- 計画どおり進んだか → KPTに反映

### 80_review.md からの抽出

- レビュー指摘の要約 → 課題セクションに反映（重要なもののみ）
- 修正完了したもの → 成果セクションに反映

### 99_history.md からの抽出

- 意思決定の記録 → KPTに反映
- 後回し判断 → 明日のTODOに反映

## KPT生成のヒント

### Keep候補
- スムーズに完了したタスク
- 外部リソース（deepwiki/WebSearch/Context7）が役立った場面
- 過去知見（memories/solutions/）が再利用できた場面
- ユーザーとのコミュニケーションがスムーズだった点

### Problem候補
- 時間がかかったタスク
- 情報不足で追加調査が必要になった場面
- 想定外の問題が発生した場面
- 手戻りが発生した作業

### Try候補
- Problemに対する具体的な改善案
- 使えそうな新しいツール・アプローチ
- プロセス改善のアイデア
- 次回のための事前準備

## 複数PJ時の構成

PJが複数ある場合:

```markdown
## 今日やったこと

### [biz-claude] 日報スキルの作成
- ...

### [biz-claude] ドキュメント整備
- ...

### [project-b] 競合分析レポート
- ...
```

PJ名はディレクトリ名を使用。

## ビジネスタスクの分類

ビジネスワークフロー（Phase 0-4）に対応した分類:

| Phase | 内容 | 日報への反映 |
|-------|------|------------|
| Phase 0: 準備 | メモリ準備、過去知見検索 | 「やったこと」の準備工程として |
| Phase 1: 調査 | 情報収集、外部リソース参照 | 「やったこと」の調査内容として |
| Phase 2: 作成 | ドキュメント・成果物の作成 | 「成果」として |
| Phase 3: レビュー | 品質確認・整合性チェック | レビュー結果を「課題」に |
| Phase 4: 完了 | 完了報告、知見保存 | 「成果」として |
