# 議事録・MTGサマリー作成

会議メモを構造化された議事録に変換し、アクションアイテムを明確にする。

## Purpose

会議の内容を効率的に整理し、参加者が次のアクションを明確に理解できる形式にまとめる。

## Key Concepts

### 議事録の構成要素

1. **会議情報**: 日時、参加者（役割付き）、目的
2. **背景・コンテキスト**: **なぜこの会議が開催されたか、前回からの経緯**
3. **アジェンダ**: 議題一覧
4. **議論内容**: 各議題の詳細・**賛否両論・検討した代替案**
5. **決定事項**: 合意された内容・**なぜその結論に至ったか**
6. **アクションアイテム**: 誰が・何を・いつまでに
7. **次回予定**: 次の会議の日程・議題
8. **用語集**: 会議中に出た専門用語・略語・固有名詞の定義（※該当がある場合）

### 10年後耐久性の原則（CRITICAL）

**「10年後に読み返しても、なぜその判断に至ったかが分かる」議事録を書く。**

- 参加者には役割を併記（「田中」→「田中（開発リーダー）」）
- 決定事項には必ず理由を付ける（「Aに決定。理由: コスト面でBより優位」）
- 却下された案も残す（「Bも検討したが、〇〇の理由で不採用」）
- 背景・前提を省略しない（初見の人が読んでも議論の流れが分かるように）

### アクションアイテムの形式

```
- [ ] [担当者（役割）]: [タスク内容] (期限: YYYY-MM-DD)
```

## Application

### Step 1: 会議メモの読み取り

ユーザーから提供された会議メモを読み取り:
- 参加者を特定
- 議題を抽出
- 議論の流れを把握

### Step 2: 構造化

以下のテンプレートに整理:

```markdown
# [会議名] 議事録

## 会議情報
- 日時: YYYY-MM-DD HH:MM - HH:MM
- 場所: [場所/オンライン]
- 参加者:
  - A（[役割・所属]）
  - B（[役割・所属]）
  - C（[役割・所属]）
- 記録者: [名前]

## 背景・コンテキスト
[この会議が開催された背景。前回からの経緯や、議論の前提となる状況を記載。
10年後に読む人が「なぜこの議題が上がったのか」を理解できるように書く。]

## アジェンダ
1. [議題1]
2. [議題2]
3. [議題3]

## 議論内容

### 1. [議題1]
**背景**: [なぜこの議題が上がったか]

**概要**: [議論の概要]

**主な意見**:
- Aさん（[役割]）: [意見と根拠]
- Bさん（[役割]）: [意見と根拠]

**検討した選択肢**:
- 案1: [内容] → [メリット/デメリット]
- 案2: [内容] → [メリット/デメリット]

**結論**: [決定事項]
**決定理由**: [なぜこの結論に至ったか]

### 2. [議題2]
...

## 決定事項
| 決定内容 | 理由 | 関連議題 |
|---------|------|---------|
| [決定1] | [なぜそう決めたか] | 議題1 |
| [決定2] | [なぜそう決めたか] | 議題2 |

## アクションアイテム
- [ ] A（[役割]）: [タスク1] (期限: YYYY-MM-DD)
- [ ] B（[役割]）: [タスク2] (期限: YYYY-MM-DD)

## 次回予定
- 日時: YYYY-MM-DD HH:MM
- 議題: [予定議題]

## 用語集（該当がある場合）
| 用語 | 説明 |
|------|------|
| [略語・専門用語] | [定義] |
```

### Step 3: 文脈の補強

AskUserQuestionで以下を確認:
- 議論の背景・経緯で補足すべきことはないか
- 決定事項の理由で追記すべきことはないか
- 会議中に出た専門用語・略語で定義が必要なものはないか

### Step 4: 確認

- アクションアイテムの担当者（役割付き）・期限が明確か確認
- 不明点があればAskUserQuestionで確認
- **10年後チェック**: 前提知識ゼロの人が読んで議論の経緯と判断理由を理解できるか

## Examples

### 例: 週次ミーティングの議事録化

```
入力（生のメモ）:
---
3/8 週次MTG
田中、佐藤、山田
・プロジェクトAの進捗確認
  → 予定通り。来週リリース予定
  → 田中さんがテスト完了させる
・新機能Bの検討
  → 佐藤さんが要件まとめる
  → 次回MTGで決定
・その他
  → 来週は水曜10時に変更
---

出力:
# 週次ミーティング 議事録

## 会議情報
- 日時: 2026-03-08
- 参加者:
  - 田中（開発リーダー）
  - 佐藤（プロダクトマネージャー）
  - 山田（QAエンジニア）

## 背景・コンテキスト
プロジェクトA（社内業務効率化ツール）のリリースに向けた週次進捗確認。
並行して次期機能Bの検討を開始する段階にある。

## 議論内容

### 1. プロジェクトAの進捗確認
**背景**: 3月中旬のリリース目標に向けた最終フェーズ。
**概要**: 進捗状況の確認。
**結論**: 予定通り進行中。来週リリース予定。

### 2. 新機能Bの検討
**背景**: ユーザーから要望が多かった機能の実現可能性検討。
**概要**: 新機能の方向性を議論。
**結論**: まず要件を整理し、次回MTGで実装可否を判断する。
**決定理由**: 技術的な実現可能性を確認してから着手すべきと判断。

## 決定事項
| 決定内容 | 理由 |
|---------|------|
| プロジェクトAは来週リリース | テスト完了次第、予定通り進行 |
| 新機能Bは要件整理後に判断 | 技術検証が先に必要 |

## アクションアイテム
- [ ] 田中（開発リーダー）: テスト完了 (期限: 2026-03-14)
- [ ] 佐藤（PM）: 新機能Bの要件まとめ (期限: 2026-03-14)

## 次回予定
- 日時: 2026-03-12 (水) 10:00
- ※通常は金曜開催だが、来週は水曜に変更
```

## Common Pitfalls

| 落とし穴 | 結果 | 対策 |
|---------|------|------|
| アクションアイテムの担当者不明 | タスクが宙に浮く | 必ず担当者（役割付き）を明記 |
| 期限なし | フォローアップできない | デフォルトで次回MTGを期限に |
| 議論と決定の混同 | 何が決まったか不明確 | 「決定事項」セクションを分離 |
| 決定理由の省略 | 後から「なぜそうなった？」が分からない | 決定事項には必ず理由を付記 |
| 背景・コンテキストの省略 | 議事録だけ読んでも議論の流れが分からない | 会議の背景を冒頭に記載 |
| 人名だけで役割なし | 誰がどの立場で発言したか不明 | 参加者には必ず役割を併記 |
