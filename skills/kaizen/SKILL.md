# カイゼン：継続的改善

継続的改善のマインドセットを適用する。小さな反復的改善、エラーを防ぐ設計、確立されたパターンの遵守、過剰な準備の回避を提案し、品質とシンプルさを導くために自動的に適用される。

## 概要

小さな改善を、継続的に。設計によってエラーを防ぐ。うまくいく方法に従う。本当に必要なものだけを作る。

**中核原則:** 一度の大きな変更よりも、多くの小さな改善の方が効果的である。修正によってではなく、設計段階でエラーを防ぐ。

## 使用する場面

**常に適用される対象:** 業務プロセスの実装/改善、ワークフロー設計、作業手順の改善

**哲学:** 大規模な努力による完璧さではなく、段階的な進歩と予防による品質。

## 4つの柱

### 1. 継続的改善（Kaizen）

小さく頻繁な改善は、大きな成果へと積み重なる。

#### 原則

**革命的よりも漸進的に:**
- 品質を向上させる最小限の変更を行う
- 一度に1つの改善
- 各変更を検証してから次へ進む
- 小さな成功を積み重ねて勢いを作る

**常に仕事をより良くして終える:**
- 出会った小さな問題を修正する
- 作業の範囲内で改善する
- 古くなった情報を更新する
- 不要なものを見つけたら削除する

**反復的な洗練:**
- 最初の版：動くようにする → 2回目：分かりやすくする → 3回目：効率的にする
- 一度にすべてをやろうとしない

#### 実務での適用

**新しい業務を始めるとき:**
1. 動く最もシンプルなバージョンから始める
2. 改善を1つ追加する（確認ステップ、バリデーションなど）
3. 実行して検証する
4. 時間が許せば繰り返す。最初から完璧を目指さない

**業務改善時:**
- 一度に1つの課題を修正し、改善ごとに記録
- 常に業務が回る状態を保つ
- 「十分良い」時点で止める（限界効用の低下）

**振り返り時:**
- 全体の作り直しではなく段階的な改善を提案する
- 優先順位：致命的 → 重要 → あれば良い
- 完璧でなくても「以前より良い」ことを受け入れる

### 2. ポカヨケ（エラー防止）

実行時ではなく、設計段階でエラーを防ぐシステムを設計する。

#### 原則

**エラーを不可能にする:**
- チェックリストがミスを検出する
- フォーマットが間違いを防ぐ
- 不正な手順を取れないようにする
- 早期にエラーを検出する

**安全のための設計:**
- 早く、はっきり失敗する
- 分かりやすいエラーメッセージを提供する
- 正しい道を明確にし、間違った道を難しくする

**多層防御:**
1. フォーマット・テンプレート（最初から正しく）
2. チェックリスト（実行時の確認）
3. レビュー（他者の目）
4. フィードバック（改善のループ）

---

### 3. 標準化された作業

確立されたパターンに従い、うまくいく方法を文書化し、良い実践を簡単に守れるようにする。

#### 原則

**賢さより一貫性:**
- 既存の方法に従う
- 解決済み問題を再発明しない
- 明確に優れている場合のみ新しい方法を導入し、チームで合意する

**ドキュメントは仕事と共に:**
- README：セットアップと概要
- 手順書：やり方
- コメントは「何を」ではなく「なぜ」を書く

**標準を自動化:**
- テンプレート（形式）、チェックリスト（確認）、定期レビュー（品質）

---

### 4. ジャストインタイム（JIT）

今必要なものだけを作る。それ以上でも以下でもない。早すぎる準備や過剰設計を避ける。

#### 原則

**YAGNI（どうせ必要にならない）:**
- 現在の要件のみ実装する
- 「念のため」「将来必要かも」の準備は避ける
- 推測による準備は削除する

**動く最も単純なもの:**
- まずは素直な解決策から始める
- 必要になったら複雑さを追加する
- 要件が変わったら見直す

**測定してから最適化する:**
- 早すぎる最適化は禁止。最適化の前に測定する
- 変更の影響を測定し、「十分速い」を受け入れる

---

## 覚えておくこと

**Kaizen とは:** 小さな改善を継続 / 設計でエラーを防ぐ / 実証済みパターンに従う / 必要なものだけを作る

**Kaizen ではないもの:** 最初から完璧を目指す / 大規模な作り直し / 複雑すぎる仕組み / 早すぎる最適化

**マインドセット:** 今日は十分に良く、明日はもっと良く。これを繰り返す。
