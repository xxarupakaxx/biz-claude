# PRD（製品要件定義書）作成

ユーザーの要件をヒアリングし、開発チームが実装可能な形式のPRDを作成する。

## Purpose

製品要件を明確化し、ステークホルダー間の認識を統一。開発チームが正しく実装できるドキュメントを作成する。

## Key Concepts

### PRDの構成要素

1. **概要**: 製品・機能の目的
2. **背景**: なぜ必要か、解決する課題
3. **ユーザーストーリー**: 誰が・何を・なぜ
4. **機能要件**: 具体的な機能
5. **非機能要件**: パフォーマンス、セキュリティ等
6. **受入基準**: 完了の定義
7. **スコープ外**: 今回対象外の内容
8. **マイルストーン**: タイムライン

### ユーザーストーリー形式

```
[ペルソナ]として、
[機能・行動]したい、
なぜなら[理由・価値]だからだ。
```

### 受入基準（Gherkin形式）

```
Given [前提条件]
When [アクション]
Then [期待結果]
```

## Application

### Step 1: ヒアリング

以下を確認:
1. 何を作りたいか？（機能の概要）
2. 誰のための機能か？（ターゲットユーザー）
3. なぜ必要か？（背景・課題）
4. 成功の定義は？（KPI・目標）
5. 制約はあるか？（期限、技術、予算）

### Step 2: ユーザーストーリー作成

ヒアリング結果からユーザーストーリーを作成:
- 主要なユースケースを洗い出し
- 優先度を設定（Must/Should/Could/Won't）

### Step 3: PRDドラフト作成

```markdown
# [機能名] PRD

## 概要
[1-2文で機能を説明]

## 背景・目的
### 解決する課題
[現状の課題]

### 期待される効果
[機能によって得られる価値]

## ターゲットユーザー
[ペルソナの説明]

## ユーザーストーリー

### US-1: [ストーリー名]
**ユーザー**: [ペルソナ]
**ストーリー**: [ペルソナ]として、[機能]したい、なぜなら[理由]だからだ。

**受入基準**:
- Given [前提条件]
  When [アクション]
  Then [期待結果]

**優先度**: Must/Should/Could

### US-2: [ストーリー名]
...

## 機能要件

### 必須要件（Must）
- [ ] 要件1
- [ ] 要件2

### 推奨要件（Should）
- [ ] 要件1

### 希望要件（Could）
- [ ] 要件1

## 非機能要件

### パフォーマンス
- [要件]

### セキュリティ
- [要件]

### ユーザビリティ
- [要件]

## スコープ外
- [今回対象外の内容1]
- [今回対象外の内容2]

## マイルストーン
| マイルストーン | 予定日 | 内容 |
|--------------|-------|------|
| 設計完了 | YYYY-MM-DD | |
| 開発完了 | YYYY-MM-DD | |
| リリース | YYYY-MM-DD | |

## 関連ドキュメント
- [デザインドキュメント]
- [技術仕様書]
```

### Step 4: レビュー

- 要件の抜け漏れがないか
- 受入基準が明確か
- スコープが適切か
- ステークホルダーに確認

### Step 5: 知見保存

PRD作成のプロセスで得た知見をmemories/に保存。

## Examples

### 例: ユーザー登録機能のPRD

```markdown
# ユーザー登録機能 PRD

## 概要
新規ユーザーがメールアドレスとパスワードでアカウントを作成できる機能。

## 背景・目的
### 解決する課題
現状、サービス利用にはアカウントが必要だが、登録フローがない。

### 期待される効果
- 新規ユーザー獲得
- ユーザーデータの蓄積

## ユーザーストーリー

### US-1: メール登録
**ユーザー**: 新規訪問者
**ストーリー**: 新規訪問者として、メールアドレスで登録したい、なぜならすぐにサービスを使いたいからだ。

**受入基準**:
- Given ユーザーが登録ページにいる
  When メールアドレスとパスワードを入力して送信
  Then アカウントが作成され、確認メールが送信される

**優先度**: Must
```

## Common Pitfalls

| 落とし穴 | 結果 | 対策 |
|---------|------|------|
| 受入基準が曖昧 | 「完了」の認識がズレる | Gherkin形式で具体的に |
| スコープが広すぎ | 期限に間に合わない | スコープ外を明記 |
| ユーザー視点の欠如 | 使いにくい機能になる | ユーザーストーリーから始める |
