# ビジネスワークフロー詳細

ビジネスタスクを効率的かつ品質高く進めるための作業ルール。

## Pipeline Map

```
memories/ ←──────────────────────────────────────────┐
    │                                                │
    ▼                                                │
learnings-researcher ─── 過去知見を全Phaseに供給     │
    │                                                │
    ▼                                                │
Phase 0 ─→ Phase 1 ─→ Phase 2 ─→ Phase 3 ─→ Phase 4
 準備       調査       作成       レビュー    完了
                                    │          │
                               reviewer    knowledge-
                               agent       capture
                                              │
                                              └─→ solutions/
                                                   （複利ループ）
```

## Phase 0: 準備

1. システムプロンプトの`Today's date`から日付取得 → `.local/memory/YYMMDD_<task_name>/`作成
2. 05_log.md初期化、ユーザーの最初の指示を記録
3. 過去知見を検索（memories/ + solutions/ を横断）、結果を05_log.mdに記録

## Phase 1: 調査

**IMPORTANT**: 発見・試行錯誤は逐次05_log.mdに記録すること

### 1.1 過去知見参照

memories/とsolutions/からタスクに関連する過去の知見を検索:
- キーワード検索: `rg "^summary:.*keyword" .local/memories/ -i`
- タグ検索: `rg "^tags:.*keyword" .local/memories/ -i`

### 1.2 外部情報参照（必須・最低1回）

| ツール | 用途 |
|--------|------|
| **deepwiki** | GitHubリポジトリの設計・アーキテクチャ理解 |
| **WebSearch** | 一般的な情報、最新ニュース、ブログ記事 |
| **Context7** | ライブラリ/フレームワークのドキュメント |

### 1.3 調査完了ゲート

調査が十分かを確認:
- 必要な情報が収集できたか
- 不明点はないか
- 追加調査が必要な場合は継続

## Phase 2: 作成

ドキュメント・成果物を作成するフェーズ。

### 2.1 構成の検討

作成する成果物の構成を検討:
- 目的・対象読者を明確化
- セクション構成を決定
- 必要な情報を整理

### 2.2 ドラフト作成

- 構成に沿ってドラフトを作成
- 05_log.mdに作成状況を記録

### 2.3 ユーザー確認（推奨）

大規模な成果物の場合、AskUserQuestionで方向性を確認してから詳細化。

## Phase 3: レビュー

品質確認・整合性チェックを行うフェーズ。

### 3.1 セルフチェック

以下の観点で確認:
- 情報の正確性
- 論理的な整合性
- 読みやすさ・分かりやすさ
- 誤字脱字

### 3.2 サブエージェントレビュー（推奨）

Taskツールで`reviewer`エージェントを起動し、第三者視点でレビュー:
- 内容の妥当性
- 抜け漏れの確認
- 改善提案

### 3.3 修正

レビュー結果に基づいて修正。修正内容を05_log.mdに記録。

## Phase 4: 完了

### 4.1 完了報告

以下を含む完了報告を作成:
1. 作成した成果物の概要
2. 自律決定した事項
3. 残存する課題（あれば）

### 4.2 知見保存

価値ある発見・知見があった場合:
1. `knowledge-capture`スキルを使用
2. memories/にインデックスを作成
3. 詳細な解決策はsolutions/に保存

## レビューアー

### docs-reviewer（ドキュメントレビュー）

以下の観点でドキュメントをレビュー:
- 構成・構造の適切さ
- 情報の正確性・網羅性
- 読者にとっての分かりやすさ
- 文体・トーンの一貫性

### consistency-reviewer（整合性レビュー）

以下の観点で整合性をチェック:
- ドキュメント間の矛盾
- 用語の統一
- 参照の正確性

## よくあるタスクパターン

### パターン1: 引き継ぎ資料作成

```
Phase 0: メモリ準備
Phase 1: deepwikiでリポジトリ調査 → README・設定ファイル読み取り
Phase 2: 引き継ぎ資料ドラフト作成
Phase 3: レビュー・修正
Phase 4: 完了・知見保存
```

### パターン2: リサーチレポート

```
Phase 0: メモリ準備 + 過去の類似調査を検索
Phase 1: WebSearch/deepwikiで情報収集
Phase 2: レポートドラフト作成
Phase 3: 情報の正確性レビュー
Phase 4: 完了・知見保存
```

### パターン3: 会議メモ整理

```
Phase 0: メモリ準備
Phase 1: 会議メモの読み取り・理解
Phase 2: 構造化された議事録作成
Phase 3: アクションアイテム抽出・確認
Phase 4: 完了・知見保存
```

## 禁止事項

- 計画なしで成果物作成開始
- 外部情報を参照せずに進行（Phase 1必須）
- レビューをスキップして完了報告
- 05_log.md未更新で次Phase移行
- 価値ある知見を保存せずにセッション終了
