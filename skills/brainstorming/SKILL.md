# アイデアをデザインに昇華する

自然な対話を通じて、アイデアを完全な設計・仕様に仕上げる支援をする。

## Purpose

曖昧なアイデアを具体的な設計・仕様に変換し、実行可能な計画を作成する。

## Key Concepts

### ブレインストーミングの原則

1. **一度に1つの質問**: 複数の質問で圧倒しない
2. **選択式優先**: 自由回答より答えやすい選択肢を提示
3. **段階的な深掘り**: 大きな方向性から詳細へ
4. **代替案の探索**: 決定前に2-3のアプローチを提案

### Done Framework

効果的なアウトプットのために以下を明確化:
1. **「完了」とは何か？**: 成果物の具体的な姿
2. **必要なコンテキストは？**: 背景情報、制約
3. **重要な制約は？**: 守るべきルール、避けるべきこと

## Application

### Step 1: 過去知見の確認

memories/とsolutions/で類似の過去アイデア・設計を検索:
- 過去に似たプロジェクトがあったか
- 活用できる知見はあるか
- 避けるべき落とし穴はあるか

### Step 2: アイデアの理解

一度に1つずつ質問してアイデアを理解:

```
質問例:
1. 「このアイデアで解決したい課題は何ですか？」
2. 「ターゲットユーザーは誰ですか？」
3. 「成功の定義は何ですか？」
4. 「制約や条件はありますか？」
```

### Step 3: アプローチの探索

2-3のアプローチを提案し、トレードオフを説明:

```markdown
## アプローチ案

### アプローチA: [名前]
**概要**: [説明]
**メリット**:
- [メリット1]
**デメリット**:
- [デメリット1]

### アプローチB: [名前]
**概要**: [説明]
**メリット**:
- [メリット1]
**デメリット**:
- [デメリット1]

### 推奨: アプローチA
**理由**: [なぜこれを推奨するか]
```

### Step 4: 設計の提示

何を作るか理解したら、設計を200-300語のセクションに分けて提示:

```
1. 「概要をまとめました。ここまでで問題ないですか？」
   → フィードバックを反映

2. 「次に詳細を説明します。[セクション内容]」
   → フィードバックを反映

3. 繰り返し...
```

### Step 5: ドキュメント化

検証済みの設計をドキュメントに:
- 設計概要
- 決定事項
- 次のステップ

### Step 6: 知見保存

ブレインストーミングで得られた有用な知見をmemories/に保存。

## Examples

### 例: 新機能のアイデア探索

```
ユーザー: 顧客管理を効率化したい

質問1:
「現在の顧客管理で最も時間がかかっている作業は何ですか？」
- 顧客情報の検索
- フォローアップの管理
- レポート作成
- その他

→ ユーザー: 「フォローアップの管理」

質問2:
「フォローアップで具体的に困っていることは？」
- いつ連絡すべきか分からない
- 連絡履歴が分散している
- 担当者間で情報共有できていない
- その他

→ ユーザー: 「いつ連絡すべきか分からない」

アプローチ提案:
## アプローチ案

### A: リマインダー機能
自動で次回連絡日をリマインド。シンプルだが手動設定が必要。

### B: 自動スケジューリング
過去の傾向から最適な連絡タイミングを自動提案。高機能だが複雑。

### 推奨: アプローチA（まずシンプルに始める）
理由: まず運用を回し、ニーズを把握してから高機能化。
```

## Common Pitfalls

| 落とし穴 | 結果 | 対策 |
|---------|------|------|
| 最初から詳細に入る | 方向性がズレたまま進む | 大きな方向性から確認 |
| 選択肢を提示しない | ユーザーが答えにくい | 2-4個の選択肢を提示 |
| 一度に多くの質問 | 混乱・疲労 | 1つずつ質問 |
| 代替案なしで決定 | 最適解を見逃す | 必ず2-3案を比較 |
