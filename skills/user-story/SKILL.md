# ユーザーストーリー作成

Mike Cohn形式のユーザーストーリーとGherkin形式の受入基準を組み合わせ、開発チームが実装可能なストーリーを作成する。

## Purpose

ユーザーのニーズを実行可能な開発作業に変換する。成果（アウトカム）に焦点を当て、製品と開発チーム間の共通理解を確保し、テスト可能な成功基準を提供する。

## Key Concepts

### ユーザーストーリー形式（Mike Cohn）

```
[ペルソナ]として、
[行動・機能]したい、
なぜなら[理由・価値]だからだ。
```

### 受入基準（Gherkin形式）

```
Scenario: [シナリオ名]
  Given [前提条件]
  When [アクション]
  Then [期待結果]
```

### INVEST原則

良いユーザーストーリーの条件:

| 原則 | 説明 |
|------|------|
| **I**ndependent | 他のストーリーから独立 |
| **N**egotiable | 詳細は交渉可能 |
| **V**aluable | ユーザーに価値を提供 |
| **E**stimable | 見積もり可能なサイズ |
| **S**mall | 1スプリントで完了可能 |
| **T**estable | テスト可能 |

## Application

### Step 1: コンテキスト収集

以下を確認:
1. **ペルソナ**: 誰がこの機能を使うか？
2. **現状の課題**: 今どのように問題を解決しているか？
3. **期待する結果**: この機能で何が達成できるか？
4. **制約**: 技術的・ビジネス的な制約は？

### Step 2: ユーザーストーリー作成

```markdown
## US-[番号]: [ストーリー名]

**ペルソナ**: [ユーザータイプ]

**ストーリー**:
[ペルソナ]として、
[行動・機能]したい、
なぜなら[理由・価値]だからだ。

**優先度**: Must / Should / Could

**見積もり**: [ポイント or 時間]
```

### Step 3: 受入基準作成

各ストーリーに対して:

```markdown
**受入基準**:

Scenario: [正常系シナリオ]
  Given [前提条件]
  When [ユーザーアクション]
  Then [期待結果]

Scenario: [エッジケース]
  Given [前提条件]
  When [異常なアクション]
  Then [エラーハンドリング]
```

### Step 4: レビュー

- INVEST原則を満たしているか確認
- 開発チームが見積もり可能か確認
- 受入基準が明確でテスト可能か確認

## Examples

### 例: ソーシャルログイン機能

```markdown
## US-001: Googleログイン

**ペルソナ**: 初回訪問ユーザー

**ストーリー**:
初回訪問ユーザーとして、
Googleアカウントでログインしたい、
なぜなら新しいパスワードを作成・記憶せずにすぐにサービスを使いたいからだ。

**優先度**: Must

**見積もり**: 5ポイント

---

**受入基準**:

Scenario: Googleログイン成功
  Given ログインページにいる
  And Googleアカウントを持っている
  When 「Googleでログイン」ボタンをクリック
  And Googleの認証画面で承認
  Then アプリにログイン状態になる
  And オンボーディング画面に遷移する

Scenario: Google認証キャンセル
  Given ログインページにいる
  When 「Googleでログイン」ボタンをクリック
  And Googleの認証画面でキャンセル
  Then ログインページに戻る
  And エラーメッセージは表示されない

Scenario: 既存メールアドレスとの重複
  Given 同じメールアドレスで既に登録済み
  When 「Googleでログイン」で同じメールアドレスを使用
  Then 既存アカウントとリンクするか確認画面を表示
```

### 例: 検索機能

```markdown
## US-002: キーワード検索

**ペルソナ**: 登録済みユーザー

**ストーリー**:
登録済みユーザーとして、
キーワードで商品を検索したい、
なぜなら目的の商品を素早く見つけたいからだ。

**優先度**: Must

**見積もり**: 3ポイント

---

**受入基準**:

Scenario: 検索結果あり
  Given ホーム画面にいる
  When 検索ボックスに「りんご」と入力
  And 検索ボタンをクリック
  Then 「りんご」を含む商品一覧が表示される
  And 結果件数が表示される

Scenario: 検索結果なし
  Given ホーム画面にいる
  When 検索ボックスに「あいうえおかきくけこ」と入力
  And 検索ボタンをクリック
  Then 「該当する商品がありません」と表示される
  And 検索条件の変更を提案する

Scenario: 空白検索
  Given ホーム画面にいる
  When 検索ボックスを空のまま
  And 検索ボタンをクリック
  Then 検索は実行されない
  And 検索ボックスにフォーカスが当たる
```

## Common Pitfalls

| 落とし穴 | 結果 | 対策 |
|---------|------|------|
| 技術タスクをストーリーに | ユーザー価値が不明確 | 「誰のため？」を常に問う |
| ストーリーが大きすぎ | 見積もり不能、分割困難 | 1スプリント以内に収まるサイズに |
| 受入基準が曖昧 | 「完了」の認識ズレ | Given/When/Thenで具体的に |
| 実装詳細を書く | 開発の自由度が減る | What（何を）だけ、How（どう）は書かない |
