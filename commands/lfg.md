# LFG - 自律ワークフロー

Phase 0-4を自律的に実行し、タスクを完了させる。

## 使い方

```
/lfg <タスクの説明>
```

## 例

```
/lfg 競合Aの分析レポートを作成
/lfg このリポジトリの引き継ぎ資料を作成
/lfg 先週の会議メモを議事録にまとめる
```

## 実行フロー

### Phase 0: 準備（AUTO）

1. メモリディレクトリ作成: `.local/memory/YYMMDD_$TASK_NAME/`
2. 05_log.md初期化
3. 過去知見を検索

```
ユーザー指示: $ARGUMENTS
```

### Phase 1: 調査（AUTO）

1. 過去知見の参照（memories/, solutions/）
2. 外部情報収集:
   - deepwiki: リポジトリ調査が必要な場合
   - WebSearch: 一般的な情報収集
   - Context7: ライブラリ関連

3. 調査結果を05_log.mdに記録

**ゲート判定**:
- 調査が十分 → Phase 2へ（AUTO）
- 不明点あり → ユーザーに確認（USER）

### Phase 2: 作成（USER確認後）

1. 構成の検討
2. ドラフト作成
3. **重要**: 大規模な成果物の場合はユーザーに方向性を確認

**ゲート判定**:
- ユーザー承認 → Phase 3へ

### Phase 3: レビュー（AUTO）

1. セルフチェック:
   - 情報の正確性
   - 論理的整合性
   - 読みやすさ

2. 必要に応じてreviewerエージェントでレビュー

3. 修正を実施

### Phase 4: 完了（AUTO）

1. 完了報告:
   - 作成した成果物の概要
   - 自律決定した事項
   - 残存する課題

2. 知見保存:
   - 価値ある発見をmemories/に保存
   - 詳細な解決策をsolutions/に保存

## ゲート判定ルール

| タイミング | 条件 | アクション |
|-----------|------|-----------|
| Phase 1完了後 | 調査十分 | AUTO → Phase 2 |
| Phase 1完了後 | 不明点あり | USER確認 |
| Phase 2開始時 | 大規模な成果物 | USER確認後に作成 |
| Phase 3完了後 | 問題なし | AUTO → Phase 4 |
| Phase 3完了後 | 要修正 | 修正後に再レビュー |

## 変数

- `$ARGUMENTS`: ユーザーが指定したタスクの説明
- `$TASK_NAME`: タスク名（ケバブケース化したARGUMENTS）

## 注意事項

- 各Phaseで05_log.mdに進捗を記録
- 不明点はAskUserQuestionで確認（推測で進めない）
- 知見は必ず保存して次回に活かす
