# ユーザーストーリーマッピング

Jeff Pattonのフレームワークを使用して、ユーザーのワークフローに基づいてストーリーを整理し、価値あるリリースを計画する。

## Purpose

製品の全体像を視覚化し、ユーザージャーニーに沿ってストーリーを整理する。これにより、価値の高い最小限のリリース（MVP）を定義し、優先順位付けを容易にする。

## Key Concepts

### ストーリーマップの構造

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
         バックボーン（ユーザーの活動）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  活動1      活動2      活動3      活動4
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  タスク1-1   タスク2-1   タスク3-1   タスク4-1
  タスク1-2   タスク2-2   タスク3-2   タスク4-2
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 ストーリー  ストーリー  ストーリー  ストーリー  ← Release 1
───────────────────────────────────
 ストーリー  ストーリー  ストーリー  ストーリー  ← Release 2
───────────────────────────────────
 ストーリー  ストーリー  ストーリー  ストーリー  ← Release 3
```

### 用語

| 用語 | 説明 |
|------|------|
| **バックボーン** | ユーザーの主要な活動の流れ（左から右） |
| **活動** | ユーザーが達成しようとする大きな目標 |
| **タスク** | 活動を達成するための具体的な行動 |
| **ストーリー** | タスクを実現する機能（詳細度が高い） |
| **リリーススライス** | 横に切った水平線。各スライスが1リリース |

## Application

### Step 1: ペルソナとゴールの確認

1. 誰のためのマップか？（ペルソナ）
2. ユーザーの最終ゴールは？
3. どのような状況で使うか？

### Step 2: バックボーンの作成

ユーザーの活動を時系列で整理:

```markdown
## バックボーン

| 順序 | 活動 | 説明 |
|------|------|------|
| 1 | [活動名] | [ユーザーが何をしようとしているか] |
| 2 | [活動名] | [ユーザーが何をしようとしているか] |
| 3 | [活動名] | [ユーザーが何をしようとしているか] |
```

### Step 3: タスクの洗い出し

各活動の下にタスクを配置:

```markdown
## タスク分解

### 活動1: [活動名]
- タスク1-1: [具体的な行動]
- タスク1-2: [具体的な行動]

### 活動2: [活動名]
- タスク2-1: [具体的な行動]
- タスク2-2: [具体的な行動]
```

### Step 4: ストーリーの配置

各タスクの下にユーザーストーリーを配置（優先度順に上から下）:

```markdown
## ストーリーマップ

### 活動1: [活動名]
#### タスク1-1
- [ ] US-001: [ストーリー] (Must)
- [ ] US-002: [ストーリー] (Should)
- [ ] US-003: [ストーリー] (Could)
```

### Step 5: リリーススライスの定義

横線で区切り、各リリースを定義:

```markdown
## リリース計画

### Release 1: MVP（最小限の価値）
**ゴール**: [このリリースで達成すること]
**期間**: [予定期間]

含まれるストーリー:
- [ ] US-001, US-004, US-007

### Release 2: 機能拡充
**ゴール**: [このリリースで達成すること]
**期間**: [予定期間]

含まれるストーリー:
- [ ] US-002, US-005, US-008
```

## Examples

### 例: ECサイトのストーリーマップ

```markdown
# ECサイト ストーリーマップ

## ペルソナ
オンラインショッピング初心者の30代女性

## ゴール
欲しい商品を見つけて購入する

## バックボーン

| 活動1 | 活動2 | 活動3 | 活動4 |
|-------|-------|-------|-------|
| 商品を探す | 商品を選ぶ | 購入する | 受け取る |

---

## ストーリーマップ

### 活動1: 商品を探す

#### タスク: 検索する
- [x] US-001: キーワード検索 (Must) ← R1
- [ ] US-002: カテゴリ絞り込み (Should) ← R2
- [ ] US-003: 価格帯フィルター (Could) ← R3

#### タスク: 閲覧する
- [x] US-004: 商品一覧表示 (Must) ← R1
- [ ] US-005: ソート機能 (Should) ← R2

### 活動2: 商品を選ぶ

#### タスク: 詳細を見る
- [x] US-006: 商品詳細ページ (Must) ← R1
- [ ] US-007: レビュー表示 (Should) ← R2

#### タスク: カートに入れる
- [x] US-008: カート追加 (Must) ← R1
- [ ] US-009: 数量変更 (Should) ← R1

### 活動3: 購入する

#### タスク: 決済する
- [x] US-010: クレカ決済 (Must) ← R1
- [ ] US-011: コンビニ払い (Could) ← R3

---

## リリース計画

### Release 1: MVP
**ゴール**: 基本的な購入フローの完成
**含まれるストーリー**: US-001, 004, 006, 008, 009, 010

### Release 2: 利便性向上
**ゴール**: 商品を探しやすくする
**含まれるストーリー**: US-002, 005, 007

### Release 3: 支払い選択肢拡大
**ゴール**: 決済方法の多様化
**含まれるストーリー**: US-003, 011
```

## Common Pitfalls

| 落とし穴 | 結果 | 対策 |
|---------|------|------|
| 技術視点でマップを作る | ユーザー価値が見えない | 必ずユーザー行動から始める |
| 細かすぎるストーリー | マップが複雑で把握困難 | 適度な粒度を保つ |
| リリース1に詰め込みすぎ | MVPが大きくなる | 「なくても成立するか？」で判断 |
| 一度作って更新しない | 現実と乖離 | スプリントごとに見直す |
