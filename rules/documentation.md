# Documentation Rules

ドキュメント作成時に適用されるルール。

## 10年後耐久性（CRITICAL）

**すべてのドキュメントは「10年後に入社した新人が読んでも理解できる」品質で書くこと。**

### 暗黙知の明文化

- 「当然知っているはず」の情報こそ丁寧に書く
- 社内用語・略語・プロジェクト固有の言葉には必ず注釈をつける
- 人名が出たら役割・所属を併記する（例: 「田中（開発チームリーダー）」）
- チーム名・サービス名には一文の説明を添える

### 「なぜ」を書く

- 決定事項には必ず「なぜその判断に至ったか」を記録する
- 却下された代替案も残す（「AではなくBを選んだ。理由: ...」）
- 制約条件を明記する（期限、予算、技術的制約、組織的制約）
- 当時の状況・前提を書く（「2026年3月時点では〇〇という状況だったため」）

### 時間的コンテキスト

- 作成日・更新日を必ず記載
- 時間依存の情報にはその旨を明記（「2026年3月時点の情報。変更の可能性あり」）
- 参照した情報源のURLと参照日を記録

### 用語集の付与

- 3つ以上の専門用語・社内用語が出てくる場合は「用語集」セクションを設ける
- 略語は初出時にフルスペルを併記（例: 「SLA（Service Level Agreement: サービス品質保証）」）

## 構成・構造

- 目的と対象読者を最初に明確化
- 論理的な構成（概要→詳細→まとめ）
- 適切な見出しレベルの使用（H1→H2→H3）
- 長文は段落・箇条書きで分割

## 読みやすさ

- 一文は短く（60文字以内を目安）
- 専門用語には説明を付ける（→ 10年後耐久性ルール参照）
- 具体例を適宜挿入
- 視覚的要素（表、図、リスト）を活用

## 一貫性

- 用語の統一（同じ概念に異なる言葉を使わない）
- 文体の統一（です・ます調 or である調）
- フォーマットの統一（日付形式、数値形式等）

## 品質チェックリスト

作成後に以下を確認:
- [ ] 目的が明確か
- [ ] 対象読者に適切な内容か
- [ ] 論理的に構成されているか
- [ ] 誤字脱字がないか
- [ ] 情報が正確か
- [ ] 必要な情報が網羅されているか
- [ ] **10年後の新人が読んでも理解できるか**
- [ ] **「なぜ」が書かれているか（決定事項に理由があるか）**
- [ ] **専門用語・略語に説明があるか**
- [ ] **暗黙の前提が明文化されているか**
- [ ] **時間的コンテキスト（作成日・状況）が記載されているか**
