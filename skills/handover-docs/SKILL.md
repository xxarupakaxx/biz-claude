# 引き継ぎ資料作成

GitHubリポジトリやプロジェクト資料から、構造化された引き継ぎドキュメントを作成する。

## Purpose

プロジェクトの引き継ぎに必要な情報を効率的に収集・整理し、後任者が理解しやすい形式でドキュメント化する。

## Key Concepts

### 引き継ぎ資料の構成要素

1. **概要**: プロジェクトの目的・背景・**なぜこのプロジェクトが始まったか**
2. **歴史的経緯**: **主要な意思決定とその理由**（なぜこの技術を選んだか、なぜこの設計にしたか）
3. **アーキテクチャ**: システム構成・技術スタック・**各技術を選んだ理由**
4. **環境情報**: 開発環境・本番環境の設定
5. **運用情報**: デプロイ手順・監視・アラート
6. **連絡先**: 関係者・問い合わせ先・**各人の役割と責任範囲**
7. **既知の問題**: 注意点・ワークアラウンド・**なぜその回避策が必要か**
8. **用語集**: プロジェクト固有の用語・略語・社内用語の定義

### 10年後耐久性の原則（CRITICAL）

**「10年後に入社した新人が、この資料だけで全体像を把握できる」ことを目指す。**

- 「なぜ」を書く: 技術選定理由、設計判断の背景、却下した代替案
- 暗黙知を明文化: 口頭伝承されている運用ルール、暗黙の前提、「みんな知っている」情報
- 時間的コンテキスト: 「○年○月時点での状況」を明記
- 用語の定義: 社内用語・略語・サービス名には必ず説明を付ける
- 人物の役割を併記: 「田中さん」ではなく「田中（開発リーダー・認証基盤担当）」

### アンチパターン

- 情報の羅列だけで構造化されていない
- 専門用語の説明がない
- 更新日時が不明で鮮度が分からない
- **「なぜ」がなく「何」だけ書かれている**
- **当事者にしか分からない暗黙の前提がある**
- **人名だけで役割が書かれていない**

## Application

### Step 1: 情報収集

```
1. GitHubリポジトリがある場合:
   - deepwikiでリポジトリ構造を取得
   - README.mdを読み取り
   - 設定ファイル（.env.example, docker-compose.yml等）を確認

2. 既存資料がある場合:
   - 関連ドキュメントを読み取り
   - 既存の引き継ぎ資料があればベースに
```

### Step 2: 暗黙知の掘り起こし（IMPORTANT）

AskUserQuestionで以下を積極的に質問:
- 口頭でだけ共有されている運用ルールはないか
- 「やってはいけないこと」「気をつけるべきこと」はないか
- なぜこの技術スタック・設計を選んだのか
- 過去に大きなトラブルや方針変更はあったか
- 検討したが却下した代替案はあるか

### Step 3: 構造化

```
収集した情報を以下の構造で整理:

# [プロジェクト名] 引き継ぎ資料

> 作成日: YYYY-MM-DD ／ 作成者: [名前（役割）]
> 本資料はYYYY年MM月時点の情報に基づいています。

## 1. 概要
- **何をするプロジェクトか**: [一文で説明]
- **なぜ始まったか**: [背景・課題]
- **主な機能**: [箇条書き]
- **関連するサービス・プロジェクト**: [名前と関係性]

## 2. 歴史的経緯と主要な意思決定
| 時期 | 出来事・判断 | 理由・背景 |
|------|------------|-----------|
| YYYY-MM | [判断内容] | [なぜそう判断したか] |

## 3. アーキテクチャ
- **システム構成図**: [図または説明]
- **技術スタック**: [一覧]
- **なぜこの技術を選んだか**: [各技術の選定理由]
- **検討したが採用しなかった技術**: [代替案と不採用の理由]

## 4. 環境情報
- 開発環境のセットアップ手順
- 本番環境の情報
- **環境ごとの違いと理由**

## 5. 運用
- デプロイ手順
- 監視・アラート
- **「やってはいけないこと」リスト**
- **過去のトラブルと対処法**

## 6. 関係者
| 名前 | 役割・責任範囲 | 連絡先 | 備考 |
|------|--------------|-------|------|
| [名前] | [役割の説明] | [連絡先] | [知っていること] |

## 7. 既知の問題・注意点
| 問題 | 影響 | 回避策 | なぜ根本対応していないか |
|------|------|-------|---------------------|
| [問題] | [影響範囲] | [回避策] | [理由] |

## 8. 用語集
| 用語 | 説明 |
|------|------|
| [用語・略語] | [定義と文脈] |

## 更新履歴
- YYYY-MM-DD: 初版作成（[作成者名]）
```

### Step 4: レビュー

- 情報の正確性を確認
- 不足している情報をAskUserQuestionで確認
- 専門用語に説明を追加
- **10年後チェック**: 前提知識なしの新人が読んで理解できるか
- **「なぜ」チェック**: 決定事項に理由が書かれているか
- **暗黙知チェック**: 口頭伝承が残っていないか

### Step 5: 知見保存

価値ある発見（効率的な調査方法、便利なツール等）があれば memories/ に保存。

## Examples

### 例1: GitHubリポジトリからの引き継ぎ資料作成

```
ユーザー: このリポジトリの引き継ぎ資料を作って
         https://github.com/company/project

実行:
1. deepwikiでリポジトリの概要・設計を取得
2. README.md、package.json、設定ファイルを読み取り
3. 引き継ぎ資料テンプレートに情報を整理
4. 不明点をユーザーに確認
5. 最終版を作成
```

## Common Pitfalls

| 落とし穴 | 結果 | 対策 |
|---------|------|------|
| deepwiki未使用 | 設計意図を見落とす | 必ずdeepwikiで全体像を把握 |
| 環境情報の省略 | セットアップで躓く | .env.example等を必ず確認 |
| 更新日時なし | 鮮度が不明 | 作成日・更新日を明記 |
| 「なぜ」の省略 | 判断の再現ができない | 技術選定理由・設計判断の背景を必ず記録 |
| 暗黙知の未記録 | 口頭伝承が途絶えると情報喪失 | AskUserQuestionで積極的に掘り起こす |
| 人名だけで役割なし | 離職後に誰が何を知っていたか不明 | 名前には必ず役割・責任範囲を併記 |
