# 引き継ぎ資料作成

GitHubリポジトリやプロジェクト資料から、構造化された引き継ぎドキュメントを作成する。

## Purpose

プロジェクトの引き継ぎに必要な情報を効率的に収集・整理し、後任者が理解しやすい形式でドキュメント化する。

## Key Concepts

### 引き継ぎ資料の構成要素

1. **概要**: プロジェクトの目的・背景
2. **アーキテクチャ**: システム構成・技術スタック
3. **環境情報**: 開発環境・本番環境の設定
4. **運用情報**: デプロイ手順・監視・アラート
5. **連絡先**: 関係者・問い合わせ先
6. **既知の問題**: 注意点・ワークアラウンド

### アンチパターン

- 情報の羅列だけで構造化されていない
- 専門用語の説明がない
- 更新日時が不明で鮮度が分からない

## Application

### Step 1: 情報収集

```
1. GitHubリポジトリがある場合:
   - deepwikiでリポジトリ構造を取得
   - README.mdを読み取り
   - 設定ファイル（.env.example, docker-compose.yml等）を確認

2. 既存資料がある場合:
   - 関連ドキュメントを読み取り
   - 既存の引き継ぎ資料があればベースに
```

### Step 2: 構造化

```
収集した情報を以下の構造で整理:

# [プロジェクト名] 引き継ぎ資料

## 1. 概要
- 目的・背景
- 主な機能

## 2. アーキテクチャ
- システム構成図
- 技術スタック

## 3. 環境情報
- 開発環境のセットアップ手順
- 本番環境の情報

## 4. 運用
- デプロイ手順
- 監視・アラート

## 5. 連絡先
- 関係者一覧

## 6. 既知の問題・注意点
- 既知のバグ
- ワークアラウンド

## 更新履歴
- YYYY-MM-DD: 初版作成
```

### Step 3: レビュー

- 情報の正確性を確認
- 不足している情報をAskUserQuestionで確認
- 専門用語に説明を追加

### Step 4: 知見保存

価値ある発見（効率的な調査方法、便利なツール等）があれば memories/ に保存。

## Examples

### 例1: GitHubリポジトリからの引き継ぎ資料作成

```
ユーザー: このリポジトリの引き継ぎ資料を作って
         https://github.com/company/project

実行:
1. deepwikiでリポジトリの概要・設計を取得
2. README.md、package.json、設定ファイルを読み取り
3. 引き継ぎ資料テンプレートに情報を整理
4. 不明点をユーザーに確認
5. 最終版を作成
```

## Common Pitfalls

| 落とし穴 | 結果 | 対策 |
|---------|------|------|
| deepwiki未使用 | 設計意図を見落とす | 必ずdeepwikiで全体像を把握 |
| 環境情報の省略 | セットアップで躓く | .env.example等を必ず確認 |
| 更新日時なし | 鮮度が不明 | 作成日・更新日を明記 |
